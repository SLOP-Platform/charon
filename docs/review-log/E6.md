# Review-log fragment — E6: wire the native work-engine end-to-end

Ticket E6 (`feat/engine-integration`, depends_on E4). Owns `src/charon/cli.py`,
`src/charon/validate.py`, `tests/test_engine_e2e.py`. Reconcile-before-code (house rule,
ADR-0010 §"Reconcile in REVIEW-LOG before code").

## Goal
Make the OPT-IN work-engine runnable end-to-end and prove it: a unit plan → board →
scheduler (warm-ACP units through the fenced `coordinator.run`) → propose-default land →
top-level end-product validation. Plus the D12 validator wired to the real top-level
acceptance captured by intake (ADR-0008), replacing the unit-level stopgap at the
integration boundary.

## Decisions

- **D-E6-1 — opt-in `charon work` command, engine imported LAZILY.** `engine.*`,
  `land`, `intake` are imported *inside* the command body, never at cli module top. The
  gateway path (`proxy_server`/`gateway`/`service.app`) therefore still transitively
  imports nothing from `engine.*` — the E0 anti-dilution boundary guard
  (`test_boundary.py::test_gateway_path_does_not_import_engine_transitively`) stays green.
  `charon work` is one more opt-in orchestrator consumer on the shared core (D001/D011),
  never on the gateway hot path.

- **D-E6-2 — drive each unit through the SINGLE fenced `coordinator.run`.** The command
  wires `engine.scheduler.CoordinatorRunner` (the default `FencedRunner` that reuses
  `coordinator.run` exactly as `api` does) — the scheduler is never a second, unfenced
  dispatch path (D008 / ADR-0010 D2, DTC Lens-2). Autonomy defaults to **L1** so a
  completed unit keeps its changes + advances `lkg`, which the land gate and the
  integration assembly both read. The escape-scan still rejects + rolls back per unit.

- **D-E6-3 — a shared base repo + per-unit LINKED worktrees, so there is a real
  integrated result.** Each unit runs in its own `git worktree add`-linked tree off ONE
  base repo (`api._prepare_repo`'s real-repo shape, nested `…/<id>/repo` so the fence's
  `guard_dir = worktree.parent` is unit-unique). Because the units share one object
  store, the integrated end-product can be assembled from each DONE unit's blessed `lkg`
  commit. Worktrees are created on the scheduler's MAIN thread (`_claim` is serial), so no
  `git worktree` concurrency lock is needed. With no `--repo`, a fresh sandbox base repo
  is created — the demo path.

- **D-E6-4 — propose-default land per unit, unchanged (D3).** Each DONE unit is run
  through `land.land_unit(ledger, unit.owns)` and its PROPOSE/HOLD verdict + diff
  (`changed_files`) is reported. The command never merges and never auto-opens PRs by
  default — a human merge is still the only thing that lands. A future `auto_land` flag is
  read generically (see D-E6-5) but its trust-extending behavior stays gated (ADR-0010 D5).

- **D-E6-5 — engine options read GENERICALLY from config.** `charon work` reads an engine
  options mapping from `<state-dir>/engine.json` (optional) overlaid by CLI flags, then
  forwards it to `select_limiter(...)` (capacity policy/caps) and the `Scheduler`
  constructor via `.get()` lookups. New options (e.g. `auto_land`, AIMD tuning) are added
  by writing a key — cli.py does not change. This keeps cli.py off the critical path for
  later engine tickets, as the ticket requires. (Does NOT edit `config.py` — that file is
  another ticket's; the loader lives in cli.py, reusing only the public config-dir
  convention.)

- **D-E6-6 — the D12 validator runs ONCE on the integrated result against the TOP-LEVEL
  product acceptance.** `validate.py` gains `top_level_checks(product_acceptance)` (extract
  the executable commands captured as inline-code spans by intake's
  `## Product acceptance` section) and `validate_product(product_acceptance, worktree)`,
  which reuses the existing `validate()` quality gate. It stays a QUALITY gate, NOT a trust
  boundary (ADR-0007 D12). On fail it holds and surfaces a `fix_proposal` (a fix-unit the
  operator can spawn) — never a silent pass; an acceptance with no executable command
  falls through to `validate([])` → held for human review. **The existing
  `validate(checks, worktree)` signature is preserved unchanged** because `decompose.py`
  (not owned here) calls it — the wiring is additive.

## Scope / boundary self-check
Only `cli.py`, `validate.py`, `tests/test_engine_e2e.py` are touched (+ this fragment).
`engine/*`, `intake.py`, `coordinator.py`, `land.py` are imported, never edited. Privileged
core stays stdlib-only; no secrets. Gate green every commit.
